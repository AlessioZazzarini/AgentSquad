# Architecture

How AgentSquad works, why it's built this way, and how the pieces connect.

## System Overview

```
User creates GitHub issues with acceptance criteria
        ↓
/orchestrate triages issues → creates .tasks/<id>/ dirs
        ↓
conductor.sh runs one idempotent tick (--once, --loop, --dry-run)
        ↓
    ┌─── The Tick ────────────────────────────────────────┐
    │                                                      │
    │  1. Finalize:  ready-for-review → push + PR          │
    │  2. Reviews:   pr-created + pr-review.md → review    │
    │  3. Approve:   review-ready → approved (policy)      │
    │  4. Merge:     approved → merged (close-task.sh)     │
    │  5. Health:    check workers (warn/kill stuck)        │
    │  6. Spawn:     loop until at capacity or no tasks    │
    │     ┌─────────────┐  ┌─────────────┐  ┌──────────┐  │
    │     │ git worktree │  │ git worktree │  │   ...    │  │
    │     │  (tmux win)  │  │  (tmux win)  │  │          │  │
    │     │ Claude Code  │  │ Claude Code  │  │          │  │
    │     └──────────────┘  └──────────────┘  └──────────┘  │
    │  7. Summary:   notify cycle results                  │
    │                                                      │
    └──────────────────────────────────────────────────────┘
        ↓ (if --loop: sleep interval, repeat)
```

## Layer Architecture

### Layer 1: Compliance Enforcement (Hooks)

The foundation. These hooks run regardless of what's happening above.

| Hook | Type | File | Purpose |
|------|------|------|---------|
| Guardrails | PreToolUse | `hooks/guardrails.sh` | Blocks destructive operations (DB drops, force-push, rm -rf source, secret leaks) even when `--dangerously-skip-permissions` is active |
| Loop Stop | Stop | `hooks/loop-stop.sh` | 7-point compliance checklist. Prevents premature stopping by detecting rationalization phrases. 3-layer session isolation (env var → state file → session ID) |
| Loop Setup | — | `hooks/loop-setup.sh` | Creates state file with iteration tracking, completion promise, session binding |
| Session Summary | Stop | `hooks/session-summary.js` | Debounced (30s) transcript parser. Writes 16 metrics per session to JSONL |

**Why hooks?** They're the only mechanism that runs even in `--dangerously-skip-permissions` mode. A worker can't bypass guardrails or skip the compliance checklist.

### Layer 2: Methodology (Templates)

The 3-file system for structured autonomous work.

| File Type | Purpose | Example |
|-----------|---------|---------|
| HOW file (`loop-*.md`) | Execution instructions, phases, gates | "Phase 1: Research. Phase 2: Implement. Phase 3: Test." |
| WHAT file (`skeleton-*.md`) | Output document template | Plan document, merge report, investigation report |
| LOG file (`execution-log-*.md`) | Timestamped trace of every action | "14:30 Read app.py (120 lines). 14:31 Found bug on line 45." |

Three task types:
- **Plan** — design without coding. Output: debate-hardened plan document.
- **Implement** — build from a plan. Output: code + tests + merge report.
- **Debug** — investigate and fix. Output: root cause analysis + fix plan.

**Why 3 files?** Separation of concerns. The HOW file is reusable (template). The WHAT file is the deliverable. The LOG file is for observability. A human reviewer reads the WHAT; the LOG is for debugging the worker.

### Layer 3: Task Repository (`.tasks/`)

File-based coordination. No database, no API, no IPC.

```
.tasks/
  _queue.md                    # Priority-ordered task list
  _interfaces/                 # Shared specs for all workers
  _completed/                  # Archived tasks
  migrate-auth/
    status.json                # State machine (jq-only updates)
    acceptance-criteria.md     # Worker's contract
    execution-log.md           # Real-time progress
    environment.md             # Optional: env-specific instructions
    .worker-prompt.md          # Auto-generated by spawn-worker.sh
```

**Why files, not a database?** Workers are Claude Code sessions. They read files natively. No SDK, no API client, no connection pooling. Status updates go through `update-status.sh` (jq-based) to prevent malformed JSON.

### Layer 4: Worker Management (Scripts)

| Script | Purpose |
|--------|---------|
| `conductor.sh` | The single orchestration engine (tick-based, all modes) |
| `spawn-worker.sh` | Launch Claude Code in tmux with a dynamically built prompt |
| `check-workers.sh` | JSON output of active workers and their health |
| `update-status.sh` | Safe jq-based status.json mutations |
| `notify.sh` | Webhook notifications on status transitions |
| `lib/config.sh` | Shared config reader for `.claude/agentsquad.json` |
| `lib/worktree.sh` | Git worktree create/cleanup for parallel isolation |

### Layer 5: Commands (Slash Commands)

User-facing Claude Code commands installed in `.claude/commands/`.

| Command | Purpose |
|---------|---------|
| `/loop-start` | Start an autonomous loop with compliance enforcement |
| `/loop-cancel` | Cancel active loop |
| `/taskify` | Convert GitHub issue → task files |
| `/orchestrate` | Multi-issue dependency-aware orchestration |
| `/cleanup` | Archive task, close issue |
| `/finish` | Commit, push, update issues, update docs |

### Layer 6: Agent Specialists (Optional)

Domain-specific agents with paired skills. Users define their own.

```
.claude/agents/
  product.md      # UI/frontend specialist
  systems.md      # Backend/API specialist
  qa.md           # Testing specialist
  architect.md    # Design/architecture specialist
```

**Why generic archetypes?** The Roger originals (val-voice, sam-systems, pete-product, quinn-qa) were too project-specific. Generic archetypes with `<!-- CUSTOMIZE THIS -->` markers let users fill in their own domain knowledge.

### Layer 7: Packs (Optional Add-ons)

| Pack | What it adds |
|------|-------------|
| `github` | Label management, `close-task.sh` |
| `collab` | Cross-model collaboration bridge (think/build/debug modes) |
| `vercel` | Preview deploy waiting, E2E on previews |
| `supabase` | Branch database creation |
| `notifications` | Slack + Telegram webhook scripts |

## Key Design Principles

### 1. Files over APIs
Every piece of state is a file. Workers read files. Status is a JSON file. Logs are markdown files. Manifests are JSON files. No database, no message queue, no IPC.

### 2. Hooks are the safety net
Even with `--dangerously-skip-permissions`, guardrails and compliance still run. This is the only reliable enforcement mechanism for autonomous agents.

### 3. Fresh sessions prevent context rot
Each issue gets its own Claude Code session. Context from Issue A never pollutes Issue B. More expensive but dramatically more reliable.

### 4. Orchestrators own isolation, workers own code
Workers are dumb: they read a prompt, write code, run tests. Orchestrators are smart: they manage branches, worktrees, dependencies, manifests, PRs. Clean separation.

### 5. Battle-tested patterns only
Every hook, every timing constant (8-second sleep), every banned phrase in the compliance checklist comes from a real production failure. See `docs/learnings.md` for the 24 lessons.

## Data Flow: Orchestration Run

```
1. /orchestrate (triage only)
   ├─ Fetch issues via `gh issue list --label squad:ready`
   ├─ Parse depends-on: #N from bodies
   ├─ Topological sort (detect cycles)
   ├─ Create .tasks/<task-id>/ directories with status.json
   ├─ Apply squad:queued labels
   └─ Call conductor.sh --once

2. conductor.sh (the single engine)
   ├─ Acquire flock tick lock (one tick at a time)
   │
   ├─ Step 1: cmd_finalize_all
   │   └─ ready-for-review → push branch, create PR → pr-created
   │
   ├─ Step 2: cmd_check_reviews
   │   └─ pr-created + pr-review.md exists → review-ready
   │
   ├─ Step 3: cmd_approve_ready
   │   └─ review-ready → approved (per approval policy)
   │
   ├─ Step 4: cmd_merge_approved
   │   └─ approved → merged (via close-task.sh or gh pr merge)
   │
   ├─ Step 5: cmd_health
   │   └─ Check active workers: OK / WARNING / STUCK (kill)
   │
   ├─ Step 6: cmd_spawn_all (loops until at capacity)
   │   ├─ Find next ready task with deps met
   │   ├─ create_worktree() → .tasks/worktrees/<task-id>/
   │   ├─ spawn-worker.sh (in worktree, via tmux)
   │   └─ Repeat until MAX_WORKERS or no ready tasks
   │
   ├─ Step 7: cmd_cycle_summary → notification
   │
   └─ Release flock, exit (or sleep + repeat if --loop)

3. status.json tracks task state throughout:
   ├─ All updates via update-status.sh (jq-based)
   ├─ Dependency checks prevent premature spawning
   └─ Cascade blocking: failed dep → block dependent tasks
```

## Configuration

```json
// .claude/agentsquad.json (created by `agentsquad init`)
{
  "project": "MyApp",
  "description": "...",
  "commands": {
    "build": "pytest",
    "test": "pytest",
    "e2e": "",
    "lint": "ruff check ."
  },
  "mainBranch": "main",
  "tasksDir": ".tasks",
  "maxWorkers": 3
}
```

Read by `core/scripts/lib/config.sh`, which exports env vars with fallback defaults. All scripts source this.

## Evolution

This system evolved through 3 phases:

1. **Roger v1** (2026-02 → 2026-03): Bug-fixing only. Sequential execution. Single Claude session. `.bugs/` directory. Named agents (Salvo, Omero, Ciro).

2. **Roger v2** (2026-03 → 2026-04): Added Taskmaster compliance (stop hook), guardrails, session metrics, cross-model collaboration (Codex bridge), parallel tmux workers. All still Roger-specific.

3. **AgentSquad v0.1** (2026-04-01): Extracted from Roger. Generalized to any project. Added CLI scaffolder, packs system, config reader, git worktrees for parallel isolation. Pressure-tested by Codex, validated by E2E test on Python Flask project.
