# Tasks

## Task Repository

Tasks live in `.tasks/` (configurable via `AGENTSQUAD_TASKS_DIR`).

```
.tasks/
  _interfaces/          # Shared documentation for all workers
  _completed/           # Archived completed tasks
  migrate-auth/
    status.json         # Machine-readable state
    acceptance-criteria.md  # Definition of done
    execution-log.md    # Worker progress log
    environment.md      # Optional: environment-specific instructions
    screenshots/        # Optional: visual evidence
    .worker-prompt.md   # Auto-generated by spawn-worker.sh
  add-search/
    status.json
    acceptance-criteria.md
    ...
```

## status.json Schema

```json
{
  "status": "ready",
  "complexity": "medium",
  "priority": "P1",
  "type": "implement",
  "attempts": 0,
  "branch": "",
  "github_issue": "",
  "pr_url": "",
  "preview_url": "",
  "blocked_reason": "",
  "updated_at": "2026-04-01T12:00:00Z"
}
```

### Status Values

| Status | Meaning |
|--------|---------|
| `ready` | Task is defined, waiting to be picked up |
| `investigating` | Worker is reading code and forming hypotheses |
| `implementing` | Worker is writing code |
| `testing-local` | Worker is running build and tests |
| `testing-preview` | Worker is testing on a preview deployment |
| `ready-for-review` | Worker is done: code written, tests pass. Orchestrator handles push + PR. |
| `pr-created` | Orchestrator has pushed branch and created PR |
| `review-ready` | PR exists + pr-review.md written. Awaiting approval. |
| `approved` | Human (manual) or system (auto) approved. Ready to merge. |
| `merged` | PR merged to main. Task archived. |
| `blocked` | Worker hit a blocker and stopped |

### Updating Status

Always use the script:

```bash
bash scripts/agentsquad/update-status.sh <task-id> <field> <value>
```

Never edit `status.json` directly. The script handles:
- Type coercion (numbers, booleans, strings)
- Timestamp updates
- Path traversal validation
- Notification triggers on status changes

## Task Lifecycle

```
ready → investigating → implementing → testing-local → ready-for-review
  → pr-created → review-ready → approved → merged
                             ↘ blocked (max attempts or unresolvable issue)

Manual mode: human sets "approved" via update-status.sh
Auto mode: conductor sets "approved" after CI green + policy check
Paused mode: nothing merges (global kill switch)

Note: The worker stops at `ready-for-review`. The orchestrator (or human) then
pushes the branch, creates the PR, and sets `pr-created`. The conductor checks
for pr-review.md to promote to `review-ready`, then applies the approval policy.
```

1. **Create** task files (manually or via orchestration)
2. **Spawn** a worker via `spawn-worker.sh`
3. **Monitor** via `check-workers.sh` and `execution-log.md`
4. **Review** the PR when status reaches `ready-for-review`
5. **Close** via `close-task.sh` (merges PR, archives task, closes issue)

## Interface Documents

Place shared documentation in `.tasks/_interfaces/`. These are read by all workers to understand system architecture:

```
.tasks/_interfaces/
  auth-system.md        # How auth works
  database-schema.md    # Current schema
  api-contracts.md      # API request/response formats
```

Workers are instructed to read interface docs before starting work.
